
# Worklog

## 2026-04-29 — Voice Input / Audio Output Improvements

### Summary
Added card-level audio tracking, auto-playback of cached audio responses, and manual audio replay capability to the SBC handheld client. Addresses the gap between capability declaration and actual implementation for voice/audio features.

### Changes Made

#### `oi_client/app.py`
- **Audio tracking**: Added `_response_audio` dict to map response IDs → WAV file paths
- **Recording state**: Renamed `_audio_buffer`/`_audio_stream_id` to `_recording_chunks`/`_recording_stream_id` for clarity
- **Cache handling**: `audio.cache.put_*` commands now build per-response audio files and store them
- **Auto-playback**: Cached audio automatically plays when received via `audio.cache.put_end`
- **Manual replay**: Pressing 'A' in CARD mode replays the last response's audio (if available)
- **Audio command handling**: Implemented `audio.play` command to play cached audio by response ID
- **UI hints**: CARD mode hint now only shows "A=Replay" when audio is actually available
- **Recording indicator**: RECORDING mode now shows in the hint list

#### `oi_client/renderer.py`
- **`draw_recording_indicator()`**: New method that draws a pulsing red dot in the top-right corner when actively recording (triggered from app loop during UIMode.RECORDING)

### Design Decisions

1. **Per-response audio storage**: Instead of a single "latest" WAV file that gets overwritten, each response ID gets its own file. This allows multiple cards in a session to each have their own replayable audio.

2. **Auto-play on receive**: The gateway's `audio.cache.put_end` triggers immediate playback, matching the expected UX where the agent's spoken response plays as soon as it's received.

3. **Replay on 'A'**: In CARD view, pressing the confirm button replays audio if available. The button hint dynamically reflects this ("A=Replay" only shown when there's audio to replay).

4. **No cleanup of temp files on every chunk**: Old temp files are no longer cleaned up on every `put_end` — they're only cleaned up on app shutdown. This reduces disk I/O and avoids race conditions during rapid successive responses.
